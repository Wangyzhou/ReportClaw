## 二、Agent Team 定义（核心交付物）

### 2.1 Coordinator Agent（协调员）

**身份 soul.md：**
```markdown
# Soul — Report Coordinator

## Identity
- 角色：报告写作项目协调员
- 职责：接收用户需求，拆解为子任务，调度其他Agent，汇总交付

## Workflow
1. 接收用户指令（生成报告/改写报告/检索知识）
2. 判断任务类型 → 分发给对应Agent
3. 监控各Agent进度
4. 汇总结果，做最终质量检查
5. 返回给用户

## Rules
- 不直接写报告，只做调度和质控
- 每个子任务有明确的输入/输出格式
- 出现分歧时以用户指令为准
```

**可调用的 Skills：**
- `task_dispatch` — 拆解用户需求为子任务
- `quality_check` — 检查报告完整性、引用准确性
- `version_control` — 管理报告版本

### 2.2 Retriever Agent（检索员）

**身份 soul.md：**
```markdown
# Soul — Knowledge Retriever

## Identity
- 角色：知识库检索专员
- 职责：从知识库中精准检索相关内容，标注来源

## Workflow
1. 接收检索需求（关键词/主题/提纲章节）
2. 向量检索 + 关键词检索（混合检索）
3. 对检索结果做相关性排序
4. 输出：检索片段 + 来源标注（文档名/页码/段落ID）
5. 如覆盖度不足，主动提示协调员

## Output Format
{
  "query": "原始查询",
  "results": [
    {
      "content": "检索到的文本片段",
      "source": {
        "doc_name": "2025年AI行业报告.pdf",
        "page": 15,
        "paragraph_id": "doc_001_p15_3",
        "relevance_score": 0.92
      }
    }
  ],
  "coverage_assessment": "高/中/低",
  "missing_topics": ["可能缺失的主题"]
}
```

**Skills：**
- `hybrid_search` — 向量+关键词混合检索
- `source_tracking` — 来源标注和溯源
- `coverage_analysis` — 知识覆盖度评估

### 2.3 Writer Agent（写作员）

**身份 soul.md：**
```markdown
# Soul — Report Writer

## Identity
- 角色：报告写作专员
- 职责：基于检索结果和提纲生成高质量报告

## Workflow
1. 接收：提纲 + 检索结果 + 风格要求
2. 按章节逐一生成内容
3. 每段引用必须标注来源ID
4. 输出初稿 → 交给审查员

## Writing Rules
- 引用来源使用 [来源ID] 标注
- 保持论述逻辑连贯
- 数据必须来自知识库，不虚构
- 支持仿写模式：学习参考报告的结构和风格
```

**Skills：**
- `outline_generation` — 根据主题生成提纲
- `section_writing` — 按章节写作
- `style_mimicking` — 学习参考报告风格
- `citation_insertion` — 插入引用标注

### 2.4 Rewriter Agent（改写员）

**身份 soul.md：**
```markdown
# Soul — Report Rewriter

## Identity
- 角色：报告改写专员
- 职责：基于参考稿件进行四种模式的智能改写

## Modes
1. 数据更新：保持结构，替换旧数据为新数据
2. 视角调整：同内容，改视角/立场/受众
3. 内容扩展：基于原稿框架，补充新分析/案例/章节
4. 风格转换：正式↔通俗，中文↔英文

## Workflow
1. 接收：原稿 + 改写模式 + 具体指令
2. 解析原稿结构（标题层级、段落分布、数据点）
3. 按模式执行改写
4. 生成修订视图（diff对比）
5. 输出改写稿 + diff

## Rules
- 数据更新模式：只改数据，不改结构和论述
- 视角调整模式：内容不变，只改表达角度
- 内容扩展模式：新增内容需标注为[新增]
- 续写必须保持原稿的逻辑、数据引用风格一致
```

**Skills：**
- `structure_parser` — 解析原稿结构
- `data_update` — 数据替换
- `perspective_shift` — 视角转换
- `content_expansion` — 内容扩展
- `style_conversion` — 风格转换
- `diff_generator` — 生成修订对比视图

### 2.5 Reviewer Agent（审查员）

**身份 soul.md：**
```markdown
# Soul — Report Reviewer

## Identity
- 角色：报告审查专员
- 职责：检查报告质量，包括事实准确性、引用准确性、逻辑连贯性

## Checklist
- [ ] 引用来源是否真实存在于知识库
- [ ] 数据引用是否准确（与原文对比）
- [ ] 逻辑是否连贯
- [ ] 是否有虚构内容
- [ ] 格式是否规范
- [ ] 知识库覆盖度是否达标

## Output
{
  "verdict": "通过/需修改/不通过",
  "issues": [
    {
      "type": "引用错误",
      "location": "第3章第2段",
      "detail": "引用数据与原文不符",
      "severity": "HIGH"
    }
  ],
  "coverage_score": 0.85,
  "quality_score": 0.78
}
```

## 三、Agent 间通信协议

基于 Clawith 已有的 CollaborationService，采用 **Delegate + Consult** 模式：

```
用户请求 → Coordinator
    │
    ├─ Delegate → Retriever（检索任务）
    │     └─ 返回检索结果
    │
    ├─ Delegate → Writer/Rewriter（写作/改写任务）
    │     └─ 返回初稿
    │
    ├─ Delegate → Reviewer（审查任务）
    │     └─ 返回审查结果
    │
    └─ 如果审查不通过 → 重新 Delegate 给 Writer 修改
```

### 消息格式（Agent间传递）

```json
{
  "task_type": "retrieve | write | rewrite | review",
  "input": {
    "query": "...",
    "context": {},
    "mode": "data_update | perspective | expand | style",
    "reference_doc_id": "..."
  },
  "output_format": "markdown | json",
  "priority": "high | medium | low",
  "focus_ref": "对应的 Focus Item ID"
}
```

## 四、知识库 RAG 架构

### 4.1 文档处理管线

```
上传文档(PDF/Word/TXT/MD)
    │
    ▼
文档解析 (pypdf2 / python-docx / markdownit)
    │
    ▼
分块 (Chunk) — 按段落/页面分割，保留元数据
    │  元数据：doc_id, doc_name, page, paragraph_id, chunk_id
    │
    ▼
向量化 (Embedding) — text-embedding-3-small 或 BGE-M3
    │
    ▼
存入向量数据库 (Milvus / Chroma / Qdrant)
    + 同时存入全文索引 (Elasticsearch/PostgreSQL FTS)
```

### 4.2 检索策略

```
用户查询
    │
    ├─ 向量检索（语义相似度 top-K）
    ├─ 关键词检索（BM25 全文检索）
    │
    ▼
混合排序 (Reciprocal Rank Fusion)
    │
    ▼
Re-rank (Cohere reranker / BGE-reranker)
    │
    ▼
返回 top-N 结果 + 来源标注
```

### 4.3 溯源机制

每个 chunk 带唯一 ID：
```
{chunk_id} = {doc_id}_{page}_{paragraph_index}
```

报告中引用格式：
```
根据统计，2025年AI市场规模达到3200亿美元 [ref:doc_001_p15_3]
```

前端点击 `[ref:...]` 可以跳转到知识库原文，高亮对应段落。

## 五、核心流程（对应T5五个模块）

### 流程1：知识库管理
```
用户上传文档 → 文档处理管线 → 分块+向量化 → 入库
用户可：浏览分类、搜索、删除、更新文档
分类：政策法规 / 行业报告 / 历史报告 / 媒体资讯
```

### 流程2：知识检索（25%权重）
```
用户输入主题/提纲
  → Coordinator 分发给 Retriever
  → 混合检索 + Rerank
  → 返回带来源标注的结果
  → 用户可手动补充/排除
```

### 流程3：AI报告生成（25%权重）
```
用户输入：主题 + 可选提纲 + 可选参考报告风格
  → Coordinator → Retriever（检索知识）
  → Coordinator → Writer（基于检索结果+提纲写作）
  → Coordinator → Reviewer（审查）
  → 如不通过 → Writer 修改
  → 通过 → 交付用户
```

### 流程4：以稿写稿/改写（25%权重）
```
用户上传参考稿 + 选择改写模式
  → Coordinator 解析原稿结构
  → Coordinator → Retriever（补充新数据，如果需要）
  → Coordinator → Rewriter（执行改写）
  → 生成 diff 视图
  → Coordinator → Reviewer（审查改写质量）
  → 交付用户
```

### 流程5：版本管理（15%权重）
```
每次生成/改写自动保存版本
  → 版本列表（v1, v2, v3...）
  → 任意两个版本 diff 对比
  → 导出 Word/PDF
```

## 六、技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| **Agent框架** | Clawith (OpenClaw) | 已有代码，YAML声明式Team定义，soul/memory/skill机制，111个ClawFlows |
| **Agent通信** | Clawith CollaborationService + A2A Agent Card | Delegate/Consult/Notify三种模式 + Agent Card描述能力 |
| **自进化机制** | 借鉴 Hermes Atropos 模式 | 报告完成后自动沉淀skill file，越用越好（快40%） |
| **LLM** | GPT-4o / Claude Sonnet 4.6 | 通过 Clawith LLM 网关路由 |
| **知识库/RAG** | **RAGFlow** (InfiniFlow) | layout-aware文档解析+溯源引用+分块可视化，中文金融文档支持最好 |
| **Embedding** | BGE-M3 | 中文效果好，开源免费，RAGFlow原生支持 |
| **工具接入** | MCP协议 | 已有闻歌晴天MCP(52工具)，MCP做Agent↔工具层，A2A做Agent↔Agent层 |
| **前端** | Clawith 前端 (React) | 已有基础，加三栏模块 |
| **版本管理** | 数据库版本表 + diff算法 | 简单够用 |
| **导出** | python-docx + WeasyPrint | Word + PDF |

### 协议栈说明

```
┌─────────────────────────────┐
│   A2A (Agent-to-Agent)       │  ← Agent间任务委托、能力发现
│   Agent Card + Task Lifecycle│
├─────────────────────────────┤
│   MCP (Model Context Protocol)│ ← Agent↔工具/数据源
│   闻歌晴天MCP / Gmail / etc. │
├─────────────────────────────┤
│   HTTP/WebSocket/SSE         │  ← 传输层
└─────────────────────────────┘
```

### RAGFlow 替代方案对比

| 框架 | 溯源能力 | 中文支持 | 复杂度 |
|------|---------|---------|-------|
| **RAGFlow** (首选) | 原生溯源+可视化 | 强 | 中（独立部署） |
| LlamaIndex | Source nodes追溯，需手动格式化 | 中 | 低 |
| LangChain | RetrievalQAWithSources | 中 | 低 |
| 自研 RAG | 完全可控 | 可控 | 高（工期长） |

**建议**：先用 RAGFlow 快速跑通（周六v1.0），如果不满意再降级到 LlamaIndex 自己搭。
